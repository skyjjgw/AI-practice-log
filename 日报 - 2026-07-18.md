## 二、核心产出与关键认知

- 成功在 PyCharm 中创建 Git 仓库并首次推送至 GitHub。
- 学会日常操作流程：

```text
修改代码 → Ctrl + K 提交 → Ctrl + Shift + K 推送
```

- 掌握公司开发的分支流程：

```text
切换到 master/main → Ctrl + T 更新代码
→ 创建 feature/任务名 分支
→ 开发、提交、推送
→ 创建 PR
→ 审核并合并
```

- 了解 `Ctrl + T` 是更新当前本地分支的远程代码。
- 理解功能分支首次 Push 时，会自动在 GitHub 创建同名远程分支。
- 理解 PR 不会自动创建，需要开发者在 GitHub 手动创建。
- 理解 PR 创建后，管理员通常在 GitHub 页面审查代码、查看自动测试结果，再决定是否合并。
- 了解变基（Rebase）用于将自己的功能分支同步到最新主分支，初学阶段优先使用普通更新/合并。
- 了解远程分支与本地分支的关系：管理员或其他开发者需从远程分支创建对应的本地分支后，才能本地查看和测试代码。
- 掌握常见分支命名：

```text
feature/xxx  新功能
fix/xxx      修复问题
hotfix/xxx   紧急修复
refactor/xxx 重构
```

## 三、遇到问题与解决方案

- PyCharm 出现“无效 VCS 根映射”提示。  
  解决：重新配置正确的 Git 项目根目录并初始化 Git。

- 功能分支首次推送时卡住。  
  解决：确认 GitHub 登录授权、检查网络，并重新执行 Push；推送成功后远程自动创建同名分支。

- 混淆 Pull、Push 和 Pull Request。  
  解决：明确区分：

```text
Push：上传本地提交到远程
Pull / Ctrl + T：获取远程更新到本地
Pull Request：请求将功能分支合并到主分支
```

- 误以为管理员需要手动拉取代码才能审核。  
  解决：管理员可直接在 GitHub PR 页面查看差异和测试结果；只有需要本地测试时，才 Fetch 并签出功能分支。
